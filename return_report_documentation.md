# Flipkart GSTR Return Report Parser Documentation

This document explains the architecture and logic used to parse Flipkart's "GSTR Return Report" Excel file and transform it into a standardized JSON format.

## Overview

Unlike the Sales Report which provides raw, row-by-row transaction data, the **GSTR Return Report** is a pre-aggregated file generated by Flipkart. It contains multiple sub-tables spread across different sheets, organized into official GST sections (e.g., "Section 7(A)", "Section 12", "Section 13").

The primary goal of `gstr-parser.ts` is to extract these distinct tables, identify the relevant data rows, map state names to GST codes, and combine everything into a unified JSON structure that mirrors the official GST API payload format.

## 1. Sheet & Table Extraction Strategy

Because the GSTR Return Report is heavily formatted for human readability rather than machine parsing, standard CSV/JSON conversion fails. Tables are often floated in the middle of a sheet with merged header cells above them.

### `getSheetDataToTable`
To handle this, the parser uses a custom binary extraction method:
1.  The Excel file is read via `XLSX.read` in binary mode.
2.  `getSheetDataToTable` iterates through every raw cell ID (e.g., `A5`, `C12`) via regex matching.
3.  It constructs an intermediate dictionary mapping: `RowNumber -> ColumnID -> Value`.
4.  This isolates the visual grid of the sheet, allowing the parser to intelligently scan down the rows to find specific header titles.

### Header Identification (`ifFindThenGetKey`)
Tables do not always start on row 1. The script uses fuzzy matching across the columns of a specific row to locate standard header names (e.g., finding the exact column letter corresponding to "Aggregate Taxable Value").

## 2. Section Extraction Logic

The parser targets specific sheet names and extracts distinct datasets.

### A. B2CS Data (Business to Consumer Small)
Extracted from sheets containing **"Section 7(A)(2)"** and **"Section 7(B)(2)"**.
*   **7(A)(2) (Local/Intra-state):** Extracts "Aggregate Taxable Value" and "CGST"/"SGST". It explicitly checks the seller's home state (`myStateCode`) to determine local sales.
*   **7(B)(2) (Inter-state):** Extracts "Aggregate Taxable Value" and "IGST". It looks up the "Delivered State" column.
*   **State Matching (`findBestMatching`):** Because Flipkart's state names can vary ("Jammu & Kashmir" vs "Jammu and Kashmir"), a fuzzy word-count matching algorithm compares the text against the official `STATES` constant to ensure accurate GST state code mapping.
*   **Consolidation:** Data from both sections is merged into a single dictionary keyed by `State ID`, grouped further by `GST Total %`.

### B. Document Issue Data (Section 13)
Extracted from sheets containing **"Section 13 in GSTR-1"**.
*   Finds columns for "Invoice Series From", "To", "Total Number of Invoices", and "Cancelled if any".
*   Calculates `net_issue` (Total - Cancelled).
*   Constructs the `doc_issue` array under the `doc_det` parent node.

### C. HSN Data (Section 12)
Extracted from sheets containing **"Section 12 in GSTR-1"**.
*   Extracts HSN Number, Total Quantity, Taxable Value, and individual tax amounts.
*   Dynamically calculates the consolidated tax rate `rt` via `((IGST + CGST + SGST) / TaxableValue) * 100`.
*   Outputs standard HSN reporting metrics (`hsn_b2c`).

### D. TCS / E-Commerce Operator Data (Section 3 Supeco)
Extracted from sheets containing **"Section 3 in GSTR-8"**.
*   Extracts the "Net Taxable Value" and tax amounts specifically handled by Flipkart as the e-commerce operator.
*   The seller's/Flipkart's GSTIN is associated as the `etin` field.
*   Outputs to the `supeco.clttx` array.

## 3. Merging and Output Generation

The `processGSTRReturnReport` orchestrates all these extractions.

### Final JSON Construction
1.  **Metadata:** Resolves the `gstin` directly from the sheet headers. Calculates the financial period (`fp`) by formatting the current session string.
2.  **Tax Routing (`mergeDataWithDetails`):**
    *   Iterates through the consolidated B2CS data.
    *   Determines `sply_ty` (Supply Type). If the target state equals the user's base state, it marks it as `INTRA` and splits the tax 50/50 into `camt` and `samt`. Otherwise, it marks it as `INTER` and maps tax to `iamt`.
3.  **UI Data:** Aside from the raw JSON payload, it also computes `b2csTableData` and `totals` (Taxable Value, Integrated, Central, State taxes). This lightweight representation is returned for the front-end React components to render the visual data tables effortlessly.

The resulting object perfectly represents the pre-calculated official GST return data, ready for comparison against the raw sales calculations.
